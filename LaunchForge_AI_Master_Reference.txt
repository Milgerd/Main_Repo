1. WHAT IS THIS PROJECT?
========================================

LaunchForge AI is a production-grade, AI-powered SaaS platform that automates MVP launch workflows for startups. It handles launch orchestration, campaign asset generation, analytics, and adaptive intelligence — all in one unified system.

The name LaunchForge AI was created by the internship program. It does not exist as a real product — it is being built from scratch. The problem it solves (automating startup launches with AI) is real. The platform competes conceptually with tools like Notion, Zapier, Copy.ai, and HubSpot, but combines their capabilities into one intelligent system.

What the Platform Does for Users:
- Launch Orchestration — manages and sequences all tasks a startup needs to launch
- Campaign Asset Automation — auto-generates marketing copy, emails, and ad content using AI
- Analytics — tracks what is working and surfaces the numbers that matter
- Adaptive Intelligence — learns from data and adjusts recommendations over time

Your Role:
You are acting as software architect, engineer, and product thinker. Your deliverable is both a detailed architecture plan and the actual built platform. You are not just writing code — you are designing, building, deploying, and operationalizing the entire system.

2. THE 6-PHASE ROADMAP
========================================

Phase  Layer         Primary Outcome                                       Status
1      Foundation    Secure, deployable microservices architecture          IN PROGRESS
2      Core System   Fully operational MVP automation platform              Pending
3      Intelligence  AI-driven optimization & predictive capabilities       Pending
4      Optimization  Performance-tuned, cost-efficient production system    Pending
5      Hardening     Secure, compliant, enterprise-grade platform           Pending
6      Scale         Multi-region, ecosystem-integrated SaaS platform       Pending

Phase Descriptions:

Phase 1 — Foundation
Build the skeleton: cloud infrastructure, database design, security basics, CI/CD pipeline, environment configuration. Nothing works for users yet, but everything is ready to be built on.

Phase 2 — Core System
Build the actual product users interact with: login, dashboard, search, media management, APIs. This is the MVP — the minimum thing that works.

Phase 3 — Intelligence
Inject AI: content generation, smart recommendations, ROI predictions, feedback learning. This is where it goes from a tool to a smart tool.

Phase 4 — Optimization
Make it fast, efficient, and cost-controlled. Caching strategies, scaling, latency reduction. It works — now make it work well.

Phase 5 — Hardening
Make it trustworthy: encryption, GDPR/CCPA compliance, disaster recovery, AI safety guardrails. This separates a startup product from an enterprise product.

Phase 6 — Scale
Open it to the world: multi-region deployment, partner integrations, subscription tiers, advanced analytics. The ready-to-grow phase.

6. API ENDPOINTS (CURRENT)
========================================

Public:
  GET  /                            Returns "Server is running"
  GET  /api/health                  Returns { status: "ok" }
  POST /api/register                Creates user (email, password) -> returns success message
  POST /api/login                   Authenticates user -> returns JWT token

Authenticated (requires Bearer token):
  GET  /api/me                      Returns current user info from JWT

Admin-only (requires Bearer token + admin role):
  GET  /api/admin/test              Returns "Admin access granted"
  GET  /api/admin/users             Returns list of all users (id, email, role)
  PUT  /api/admin/users/:id/role    Updates a user's role (body: { role: "admin" | "user" })

10. ARCHITECTURE PATTERN
========================================

The backend follows a layered architecture:

  Request -> Route -> Controller -> Service -> Database

  Routes:       Define URL paths and HTTP methods, attach middleware
  Controllers:  Handle request/response, validate input, call services
  Services:     Contain business logic, interact with database
  Middleware:   Cross-cutting concerns (auth, role checks)
  DB:           Connection pool module, health checks

This separation keeps the codebase organized and testable as it grows.
